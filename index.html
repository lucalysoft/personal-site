JSHint is a community-driven tool to detect errors and potential problems in JavaScript code and to enforce your team's coding conventions. It is very flexible so you can easily adjust it to your particular coding guidelines and the environment you expect your code to execute in.

JSHint is an open-source project that is supported and maintained by the JavaScript developer community. The source code is available on GitHub.

Documentation

Usage

JSHINT is a global function. It takes two parameters.

var result = JSHINT(source, options);

The first parameter is either a string or an array of strings. If it is a string, it will be split on '\n' or '\r'. If it is an array of strings, it is assumed that each string represents one line. The source can be JavaScript or JSON.

If all of the enabled tests pass, JSHINT returns true. Otherwise, it returns false.

If false, you can use JSHINT.errors to retrieve the errors or request a complete data structure representing the "lint" by using JSHINT.data(). For more information, read the JSHint source code (it's well documented).

You can also set options in-file,

/*jshint evil: true, boss: true */

And let JSHint know what global variables it should expect,

/*global DISQUS: true, jQuery: false */

In the example above, JSHint will allow you to override DISQUS, but complain if you try to override jQuery.

How JSHint is different from JSLint

JSHint is a fork of Douglas Crockford's JSLint that is designed to be more flexible than the original. Our goal is to make a tool that helps you to find errors in your JavaScript code and to enforce your favorite coding style.

We realize that people use different styles and conventions, and we want our tool to adjust to them. JSHint will never enforce one particular convention.

JSHint is developed and supported by the JavaScript developer community and we welcome feedback from everybody who cares about the language.

Environments

You can also let JSHint know when your code is running in a specific environment. This will help it recognize specific pre-defined global variables.

Currently supported environments:

  • Browser (browser)
  • Development: console, alert, etc. (devel)
  • jQuery (jquery)
  • CouchDB (couch)
  • ES5 (es5)
  • Node.js (node)
  • Rhino (rhino)
  • Prototype.js (prototypejs)
  • MooTools (mootools)

Options

asi

If true, JSHint will tolerate the use of automatic semicolon insertion.

Automatic semicolon insertion is one of JavaScript's most controversial syntactic features. By default, JSHint requires you to use semicolons at the end of every statement but if you understand ASI and want to use them only when strictly required, this option is for you.

bitwise

If true, JSHint will prohibit the use of bitwise operators.

JavaScript supports bitwise operators but it does not have an integer type. The bitwise operators convert their operands from floating-point to integers and back, therefore they may not be as efficient as they are in other languages.

boss

If true, JSHint will allow the use of assignments inside structured elements such as if, for and while.

More often than not, you want to compare values inside loops or conditional statements and doing an assignment (= vs. ==) is a typo. By default, JSHint assumes that it is a typo, but enabling this option will ignore it so you can use the assignment operator like a boss.

curly

If true, JSHint will require that you always use blocks with structured elements such as if and while.

JavaScript allows you to omit curly braces when the body of a loop or if statement consists of a single statement. This might make your code look more confusing so there is an option to always require curly braces.

debug

If true, JSHint will allow the usage of debugger statements, which are used to set breakpoints in modern browsers.

It is probably a good idea to remove debugger statements before you ship your code.

eqeqeq

If true, JSHint will require that you use === and !== for all comparisons.

The == and != operations do type coercion before comparing. Unless you know what are you doing, you should always use === or !== when comparing to 0, '', undefined, null, false or true.

eqnull

If true, JSHint will allow the use of == null.

== null is usually used to check if something is undefined or null (that works because both null and undefined coerce to false when using ==).

evil

If true, JSHint will allow the use of eval.

The eval function provides access to the JavaScript compiler. It is sometimes useful but more often its use will open up your code for injection attacks and make debugging more difficult. It is probably a good idea to never use it unless you know what you are doing.

Note that the Function constructor is another form of eval. Also, when given string arguments (as opposed to function), setTimeout and setInterval will act as eval.

forin

If true, JSHint will allow the use of for in without hasOwnProperty.

The for in statement is used to loop through the names of properties of an object, including those inherited through the prototype chain. The method hasOwnPropery is used to check if the property belongs to an object or was inherited through the prototype chain.

immed

If true, JSHint will require immediate invocations to be wrapped in parens.

laxbreak

If true, JSHint will not check line breaks.

Automatic semicolon insertion in JavaScript can mask some serious errors, so this option checks your code for potential unsafe line breaks.

maxerr

This option is used to specify maximum number of errors before JSHint stops processing your source. Default value is 50.

newcap

If true, JSHint will require that you capitalize all constructor functions.

Constructors are functions that are designed to be used with the new statement. new creates a new object and binds it to the implied this parameter. A constructor executed without new will have its this assigned to a global object, leading to errors.

Unfortunately, JavaScript gives us absolutely no way of telling if a function is a constructor. There is a convention to capitalize all constructor names to prevent those mistakes. This option enforces that convention.

noarg

If true, JSHint will prohibit the use of arguments.caller and arguments.callee.

The arguments object is an array-like structure with an indexed property for each passed argument. arguments.callee references the currently invoking function (it is disallowed in the ES5 strict mode). arguments.caller references the function (if any) from which the current function was invoked. It is not supported in all JavaScript implementations.

noempty

If true, JSHint will prohibit the use of empty blocks.

Originally, JSLint actively prohibited the use of empty blocks (i.e. blocks without any statements). We removed that rule and made it an option.

nonew

If true, JSHint will prohibit the use of constructors for side-effects.

Originally, JSLint actively prohibited the use of constructors for side-effects. So the perfectly fine code shown below would not pass the check (note that we don't assign the result of this expression to any variable):

new UIWindow();

Since people use the above in their projects without error we made the prohibition an opt-in option.

nomen

If true, JSHint will disallow the use of initial or trailing underbars in names.

This is mostly a styling issue. Some people use _ to indicate that property is private but in JavaScript it does not make it any more private than any other property. You can achieve real private variables using closures.

onevar

If true, JSHint will allow only one var statement per function.

This is, again, mostly a styling issue.

passfail

If true, JSHint will stop on the first error it encounters.

plusplus

If true, JSHint will prohibit the use of increment and decrement operators.

Increment (++) and decrement (--) operators may lead to code that is hard to read so there is an option to prohibit their use.

regexp

If true, JSHint will disallow . and [^...] in regular expressions.

They match more material than might be expected, allowing attackers to confuse applications.

undef

If true, JSHint will require all non-global variables be declared before they are used.

If you don't explicitly declare a variable in local scope with var, JavaScript will put it in a global scope. This is usually a mistake. This option checks that all variables are defined before they are used.

sub

If true, JSHint will tolerate all forms of subscript notation.

By default, JSHint will require you to use dot-notation when accessing a property of an object (unless property name is a reserved word). Generally, it is a good idea but if you really want to use other forms, use this option.

strict

If true, JSHint will require you to use "use strict"; pragma.

Strict Mode is a new feature in ECMAScript 5 that allows you to place a program, or a function, in a "strict" operating context. This strict context prevents certain actions from being taken and throws more exceptions. For more information, see ECMAScript 5 Strict Mode, JSON, and More by John Resig.

white

If true, JSHint will check your code against strict whitespace rules.
