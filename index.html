JSHint is a community-driven tool to detect errors in JavaScript code and enforce your team's coding conventions. It is very flexible so you can easily adjust it for your particular coding guidelines and also the expected environment for your JavaScript.

JSHint is an open-source project that is supported and maintained by the JavaScript developer community. The source code is available on GitHub. GitHub.

Documentation

Usage

JSHINT is a global function. It takes two parameters.

var result = JSHINT(source, options);

The first parameter is either a string or an array of strings. If it is a string, it will be split on '\n' or '\r'. If it is an array of strings, it is assumed that each string represents one line. The source can be a JavaScript or a JSON text.

If it checks out, JSHINT returns true. Otherwise, it returns false.

If false, you can inspect JSHINT.errors to find out problems or request a complete data structure using JSHINT.data(). For more information, read JSHint source code (it has useful comments).

You can also set options in-file.

/*jshint evil: true, boss: true */

And let JSHint know what global variables it should expect.

/*global DISQUS: true, jQuery: false */

Above, JSHint will allow you to override DISQUS but complain if you try to override jQuery.

How JSHint is different from JSLint

JSHint is a fork of Douglas Crockford's JSLint that does not tyrannize your code. It is designed to detect errors that actually break your code while skipping things that, according to Crockford, “are known to contribute mistakes in projects”. In other words, JSHint is a fork of JSLint for the real world where people use different styles and conventions.

The most important part is that JSHint is developed and supported by the JavaScript developers community and not by one very opinionated person.

Environments

You can also let JSHint know when your code is running in a specific environment. This will help it to recognize specific pre-defined global variables.

Currently supported environments:

  • Browser (browser)
  • ES5 (es5)
  • Node.js (node)
  • Rhino (rhino)

Options

bitwise

If true, JSHint will prohibit the use of bitwise operators.

JavaScript supports bitwise operators but it does not have an integer type. The bitwise operators convert their operands from floating-point to integers and back and they are not as efficient as they are in other languages.

boss

If true, JSHint will allow the use of assignments inside structured elements such as if, for and while.

More often than not, you want to compare values inside loops or conditional statements and using assignment (= vs. ==) is a typo. By default, JSHint assumes that it is a typo but this option makes it look away. So you can use assignment operator like a boss.

curly

If true, JSHint will require you to always use blocks with structured elements such as if and while.

JavaScript allows you to omit curly braces when body of a loop or if statement consists of a single statement. This might make your code look more confusing so there is an option to always require curly braces.

debug

If true, JSHint will allow usage of debugger statements. This statement is used to set a breakpoint in modern browsers.

It is a good idea to not to ship your code with debugger statements in it.

devel

If true, JSHint will allow usage of logging functions such as console.log and alert.

Using console.log in the production code can be a bad idea if you expect it to be run in the Internet Explorer. IE does not support this function and fires an exception when you try to use it.

eqeqeq

If true, JSHint will require you to use === and !== for all comparisons.

The == and != operations do type coercion before comparing. Unless you know what are you doing, you should always use === or !=== when comparing to 0, '', undefined, null, false or true.

evil

If true, JSHint will allow the use of eval.

The eval function provides access to the JavaScript compiler. It is sometimes useful but more often its use will open up your code for injection attacks and make debugging more difficult. It is probably a good idea to never use it unless you know what you are doing.

Note, that the Function constructor is another form of eval. Also, when given string arguments (as opposed to function), setTimeout and setInterval will act as eval.

forin

If true, JSHint will disallow usage of for in without hasOwnProperty.

The for in statement is used to loop through the names of properties of an object, including those inherited through the prototype chain. Method hasOwnPropery is used to check if property belongs to an object or was inherited through the prototype chain.

immed

If true, JSHint will require immediate invocations to be wrapped in parens.

laxbreak

If true, JSHint will not check line breaks.

Automatic semicolon insertion in JavaScript can mask some seriours errors so this option checks your code for potentional unsafe line breaks.

newcap

If true, JSHint will require you to capitalize all constructor functions.

Constructors are functions that are designed to be used with the new prefix. The prefix creates a new object and binds it to the implied this parameter. Constructor executed without new will have its this assigned to a global object which is a mistake.

Unfortunately, JavaScript gives us absolutely no way of telling if a function is a constructor. There is a convention to capitalize all constructor names to prevent those mistakes. This option enforces that convention.

noarg

If true, JSHint will prohibit the use of arguments.caller and arguments.callee.

The arguments object is an array-like structure with an indexed property for each passed arguments. Property arguments.callee references the currently invoking function (it is disallowed in the ES5 strict mode). Property arguments.caller references the function (if any) from which the current function was invoked. It is not supported in all JavaScript implementations.

noempty

If true, JSHint will prohibit the use of empty blocks.

Originally, JSLint actively prohibited the use of empty blocks (i.e. blocks without any statements). We removed that rule and made it an option.

nonew

If true, JSLint will prohibit the use of constructors for side-effects.

Originally, JSLint actively prohibited the use of constructors for side-effects. So, the perfectly fine code shown below would not pass the check (note that we don't assign the result of this expression to any variable):

new UIWindow();

Since people use that in their projects and it works just fine, we made the prohibition an opt-in option.

nomen

If true, JSHint will disallow the use of initial or trailing underbars in names.

This is mostly a styling issue. Some people use _ to indicate that property is private but in JavaScript it does not make it any more private than any other property. You can achieve real private variables using closures.

onevar

If true, JSHint will allow only one var statement per function.

This is, again, mostly a styling issue.

passfail

If true, JSHint will stop on first error.

plusplus

If true, JSHint will prohibit the use of increment and decrement operators.

Increment (++) and decrement (--) operators may lead to code that is hard to read so there is an option to prohibit their use.

regexp

If true, JSHint will disallow . and [^...] in regular expressions.

They match more material than might be expected, allowing attackers to confuse applications.

undef

If true, JSHint will require all non-global variables to be declared before used.

If you don't explicitly declare a variable in local scope with var, JavaScript will put it in a global scope. This is usually a mistake. This option checks that all variables are defined before used.

strict

If true, JSHint will require you to use "use strict"; pragma.

Strict Mode is a new feature in ECMAScript 5 that allows you to place a program, or a function, in a "strict" operating context. This strict context prevents certain actions from being taken and throws more exceptions. For more information, see ECMAScript 5 Strict Mode, JSON, and More by John Resig.

white

If true, JSHint will check your code against strict whitespace rules.
